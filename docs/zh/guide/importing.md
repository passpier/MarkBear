# 匯入文件

大部分的內容都存放在 Word 文件、試算表或簡報中 —— 這些格式難以進行版本控制、協作或直接發布。
Pourdown 讓你能直接將這些格式匯入成可編輯的 Markdown 文件，不需要手動複製貼上就能整理、重組並匯出。

**如何匯入：** `檔案 → 匯入` → 選擇你的格式。檔案會立即以新的 Markdown 文件開啟。

> 匯入是將內容轉換成 Markdown，屬於**單向**轉換。匯出回原始格式並不會精確還原原本的版面配置。

## 各格式轉換內容

### Word（`.docx`）

- 標題（樣式 + 大綱層級）、粗體／斜體／刪除線
- 巢狀項目清單與編號清單、表格、超連結
- 內嵌圖片，會被擷取為附屬檔案並在編輯器中顯示

**限制：** 向量圖片（EMF/WMF）無法在編輯器中顯示；追蹤修訂與註解會被捨棄；
目錄會轉換為佔位符，而非重新建構。

### 試算表（`.xlsx` / `.xls` / `.ods`）

- 每個工作表都會成為一個包含完整 GFM 表格的區塊
- 自動偵測日期欄位（標題含有「date」／「日期」）並格式化為 ISO 日期
- 內嵌圖片會被擷取

**限制：** 每個工作表上限 500 列（超出時會加上提示）；圖片無法對應到特定工作表／儲存格，
因此會列在「內嵌圖片」區塊中，而非直接內嵌。

### PDF

- 依據相對於內文字型大小的比例推斷標題層級
- 段落依上到下、由左至右排序
- 透過幾何分析（連續多列共用對齊欄位）偵測表格並轉換為 GFM 表格
- 目錄（帶點狀引導線的項目）會轉換為條列清單
- 依閱讀順序擷取內嵌圖片

**限制：** 僅支援文字型 PDF —— 掃描檔／影像式 PDF 不支援；複雜的多欄版面可能會重新排序；
若表格中的換行儲存格本身是縮排／條列清單，則會回退為一般段落，而非破壞表格結構
（這是刻意採取的保守取捨）。

### PowerPoint（`.pptx`）

- 投影片標題轉換為 `#` 標題；內文轉換為段落，每張投影片為一個區塊（以 `---` 分隔）
- 保留條列／縮排層級與基本粗體／斜體格式
- 內嵌圖片會被擷取並內嵌顯示

**限制：** 動畫不會被擷取（Markdown 無法表示動畫）；向量圖片（EMF/WMF）無法顯示。

## 圖片處理

四種格式的擷取圖片都會儲存為文件旁的附屬檔案 —— 文件未儲存時位於 `assets/` 資料夾，
第一次儲存後移動到 `.md` 檔案旁的 `<檔名>.assets/` 資料夾 —— 並在編輯器中即時顯示。
`.md` 檔案本身只會儲存相對路徑，讓文件與圖片能一起移動、壓縮或提交到 git，保持可攜性。

網頁檢視元件無法呈現的向量格式（EMF/WMF，常見於 Office 匯出檔）會以
`*(unsupported image)*` 提示取代，而非顯示失效的圖片連結。

## 為什麼要匯入成 Markdown？

檔案在處理前會先轉換成 Markdown，以降低 token 用量。社群基準測試顯示，Markdown
比 JSON 大約節省 15% 的 token，而相較於原始 PDF，餵給 LLM 時可節省高達 96%。

這個做法的靈感來自微軟的 [MarkItDown](https://github.com/microsoft/markitdown)；
Pourdown 是以 Rust 獨立重新實作，而非分支（fork）或移植（port）——
詳細技術說明請參閱儲存庫中的
[`markdown-import.md`](https://github.com/passpier/Pourdown/blob/main/markdown-import.md)。
